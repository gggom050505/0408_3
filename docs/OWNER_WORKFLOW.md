# 공공곰타로덱 — 기획·디자인(오너) ↔ 개발자(동업자) ↔ 웹(URL) 적용 흐름

이 문서는 **코드 구조를 몰라도** 따라갈 수 있게, “꿈을 현실로 옮기는 순서”만 적어 둡니다.

## 당신이 하고 싶은 것 (한 줄)

**오프라인에서 앱(Flutter)으로 수정·보완 → 동업자 개발자에게 확인 → 통과하면 웹(URL)에 반영.**

## 세 단계

### 1단계: 나 — 오프라인에서 앱만 손본다

- 이 저장소(Flutter 프로젝트)에서 화면·문구·탭 동작을 바꿉니다.
- **인터넷 없이도** 타로·오라클·캡처·저장 등은 `assets/www_gggom/` 번들로 동작합니다 (`lib/main.dart` 주석 참고).
- 서버 붙는 기능(피드·상점·로그인)은 빌드할 때 Supabase 키를 넣으면 되고, 키 없이도 **게스트 둘러보기**로 화면 검증은 가능합니다.
- **실제로 어디 파일을 열지**는 [OFFLINE_EDITING.md](OFFLINE_EDITING.md) 에 표로 적어 두었습니다.

### 2단계: 동업자 개발자 — 검토·합류

- 당신이 **Git 브랜치 / ZIP / 화면 녹화** 등 팀이 쓰는 방식으로 변경분을 넘깁니다.
- 개발자가 확인할 것(예시):
  - 빌드가 되는지 (`flutter analyze`, 실기기/윈도우 실행)
  - 기획 의도와 화면이 맞는지
  - 웹에 옮길 때 **같이 옮겨야 할 에셋**(이미지·카피) 목록

### 3단계: 웹(URL)에 적용

- **Flutter 앱**과 **웹사이트**는 보통 **별도 배포**입니다. 앱에서 확정한 기획·디자인·카피를 웹 레포(Next 등)에 **반영하는 작업**이 한 번 더 필요합니다.
- 이 프로젝트에 `assets/www_gggom/` 으로 미러해 둔 자산은, 웹 `public` 과 맞추며 동기화하면 “앱·웹이 같은 느낌”을 유지하기 쉽습니다.

## 역할 정리 (헷갈릴 때)

| 누가 | 무엇을 |
|------|--------|
| **나 (오너)** | 기획·카피·우선순위, 앱에서 오프라인 검증, 개발자에게 넘길 범위 정하기 |
| **동업자 개발자** | 코드 품질·빌드·서버·웹 반영·스토어 등 기술 실행 |
| **URL(웹)** | 최종 사용자용 사이트 — 앱에서 확정한 내용을 반영하는 **또 하나의 출구** |

## 기술 구조는 짧게만

- 앱의 **시작·로그인 환경**: `lib/main.dart`, `lib/config/app_config.dart`, `lib/widgets/app_root.dart`
- **각 화면·버튼**: `lib/widgets/` 아래 파일들 (`home_screen`, `tarot_tab`, …)

구조를 외울 필요는 없고, **“어디를 고치면 화면이 바뀌는지”**는 개발자가 PR/메모로 짚어 주면 됩니다.
